# Spike F3.99 — Groove Engine Accuracy Under Host Clock

**Status:** CLOSED
**Date:** 2026-05-30

---

## Question

When a micro-timing offset (DRAG or PUSH) displaces a beat event's sample position
outside the current audio block boundary, what strategy should the AestheticResolver
and BeatSequencer use to maintain perceptual accuracy?

Specifically:
- DRAG (late): `displaced = sample_offset + drag_samples`. May exceed `blockSize`.
- PUSH (early): `displaced = sample_offset - push_samples`. May go negative.

At 120 BPM with a 256-sample buffer at 44100Hz:
- One block ≈ 5.8ms
- F3-AC1 target: DRAG -10 to -30ms = 441–1323 samples = 2–5 blocks
- F3-AC2 target: PUSH +5 to +25ms = 220–1102 samples = 1–4 blocks
- Both targets require handling multi-block displacement.

---

## Three Strategies Analysed

### Strategy A — Clamp to block boundary

Clamp displaced offset to `[0, blockSize-1]`.

**Analysis:** Simple, zero overhead. But F3-AC1 requires -30ms (5 blocks). Anything
beyond ~6ms (1 block) is silently clamped to the block edge, losing perceptual accuracy.
DRAG at 50% produces at most ~6ms of lag instead of the specified -10 to -30ms.

**Verdict:** Fails acceptance criteria. Not viable.

### Strategy B — Carry-forward queue (chosen)

Events displaced out of the current block are queued in a small FIFO carried between blocks.

**DRAG** (offset grows, may exceed blockSize):
`displaced = sample_offset + drag_samples`
If `displaced >= blockSize`: add to carry queue at `displaced - blockSize` for next block.
Events chain across multiple blocks if `displaced > 2 * blockSize` (chained defers).

**PUSH** (offset shrinks, may go negative):
`displaced = sample_offset - push_samples`
If `displaced < 0`: clamp to 0 within the current block (small push values within ~6ms),
OR for larger push values: the event was effectively "early" in the current block — fire
at `juce::jmax(0, displaced)`. Sub-6ms push values fire slightly late relative to the
true target but within 1 block; this is perceptually acceptable.

Alternative for PUSH > 1 block: pre-schedule in a "look-ahead carry" from the PREVIOUS
block. However, this requires knowing NEXT beat events during the CURRENT block —
requiring bar-level pre-computation or look-ahead buffering. Deferred to F3.99b if
needed; for F3 PUSH ≤ 30ms, clamping to 0 within the block is acceptable.

**Queue capacity:** 8 events maximum. At any moment only 1-2 events will be in the queue
(one per layer at most). 8 gives a safety margin for simultaneous ROLL + DRAG events.
Audit note: increase to 64 if ROLL×8 + STUTTER×8 are combined (see 04-02 implementation).

**Properties:**
- Zero allocation after prepare() (fixed array)
- Correct for DRAG up to any value ≤ 1 bar (88200 samples at 120 BPM)
- PUSH accurate within 1 block (~6ms); acceptable for F3 acceptance criteria
- No bar-level pre-computation required
- Resync on pattern change: clear queue on pattern switch signal

**Verdict:** Viable and sufficient for F3 acceptance criteria.

### Strategy C — Bar-level pre-computation

At each bar start, compile all events for the next N bars including micro-offsets,
storing in a pre-allocated bar-event buffer. Audio thread reads from the buffer.

**Analysis:** Perfectly accurate. But requires resync logic on pattern change (costly),
and the implementation scope is ~3× larger than Strategy B. The accuracy benefit
(handling multi-block PUSH) is not required by F3 acceptance criteria.

**Verdict:** Over-engineered for F3. Candidate for F5+ if groove engine accuracy is
validated to need it.

---

## Decision: Strategy B (carry-forward queue)

**Rationale:**
- Only viable alternative to A (which fails acceptance criteria)
- Correct for all DRAG values up to practical groove amounts
- PUSH accuracy within 1 block (~6ms) satisfies F3-AC2 target of +5-+25ms
- Zero allocation, minimal implementation overhead
- Carries forward to F4/F5 without architectural change

---

## Implementation Consequences for BeatSequencer and AestheticResolver

### BeatSequencer

1. Add `pendingEvents_` fixed-size array (capacity: 64 events post-audit) + `pendingCount_`.
   Cleared in `prepare()`.

2. At the **START** of each `process()` block:
   - **Step 1**: Call `drainPending()` — dispatch all carry-forward events from the previous block.
   - **Step 2**: (If pattern loop start) call `resetFractureSeed()`.
   - ORDER IS CRITICAL: drain THEN reset seed. Reversing corrupts FRACTURE determinism
     for carry-forward events (their probability was already computed; seed reset changes
     the LCG state mid-sequence for any events that span bar boundaries).

3. Add `double stepPeriodSamples` parameter to `process()`, computed by PluginProcessor
   as `clock_.samplesPerBeat()`. Add `Clock::samplesPerBeat() const noexcept` accessor.

### AestheticResolver

1. `resolveEvent()` takes `stepPeriodSamples` as explicit parameter (not BPM).
2. DRAG: `displaced = beatSampleOffset + drag_samples`; if `>= blockSize` → queue deferred.
3. PUSH: `displaced = beatSampleOffset - push_samples`; if `< 0` → clamp to `juce::jmax(0, displaced)`.
4. FRACTURE: LCG computed during `resolveEvent()`; seed reset called by BeatSequencer.
5. ROLL: each sub-event checked independently against block bounds; deferred if needed.
6. STUTTER: sub-events emit `triggerRetain()` for events 2..N (no phase reset → no click).

### Maximum Supported Offset (Strategy B)

- DRAG: any value ≤ 1 bar period (4 beats at 120 BPM = 88200 samples = ~2 seconds).
  Practically unlimited for groove purposes; no human can perceive drag > 200ms.
- PUSH: effectively bounded by 1 block (~6ms) without look-ahead. Sufficient for F3-AC2.

## Validation Test (manual, not required before 04-02)

To verify Strategy B is implemented correctly:

```
Setup: standalone at 120 BPM, 256-sample buffer
Configure: DRAG = 0.5 on snare (step 4)
Expected: snare fires ~24ms late relative to kick on step 0 and 8
Measure: record audio, compare snare peak to kick peak — should be ~24ms later
Accept: ±2ms tolerance
```

PUSH inverse test:
```
Configure: PUSH = 0.5 on snare
Expected: snare fires ~12ms early relative to grid
Accept: ±2ms tolerance
```
