# Spike F0.99b — Thread-Safety Model

**Status:** CLOSED
**Date:** 2026-05-28
**Author:** bidu@lavid.ufpb.br

---

## Question

Which pattern should govern `Project` model access between the audio thread and
GUI/LLM threads? And how should user-level configuration (LLM bridge toggle, API
key, endpoint, profile override) be separated from the musical project state?

---

## Options Evaluated

### lock-free-snapshot (chosen)
Audio thread reads a frozen `shared_ptr<const Project>`. GUI/LLM thread
posts mutations that atomically swap the pointer. Zero contention on audio
thread. One-block lag on mutations (5.8ms at 256-sample/44100Hz — imperceptible
for all LAYERZ mutation types). Two copies of Project in memory during swap.

### message-queue
GUI posts small delta messages via lock-free FIFO; audio thread drains per block.
No full copy needed. But "current state" must be reconstructed from message history
for the LLM bridge — fragile, complex, fights the canonical-model design.

### JUCE ValueTree
Off-the-shelf sync, undo, preset load. XML-typed; requires translation layer for
`.layerz` JSON. Hides real-time safety issues behind JUCE abstractions. LLM bridge
cannot consume ValueTree directly.

---

## Decision

**lock-free-snapshot**

Rationale (from author):
- LLM bridge is core: needs the full `Project` as a clean JSON-serializable struct
  at any moment. Snapshot = zero reconstruction needed; the snapshot IS the JSON model.
- One-block mutation lag (5.8ms) is imperceptible for LAYERZ's mutation granularity
  (step toggles, DRAG/PUSH amounts, template changes, scene switches — all human-speed).
- `getStateInformation()` / `setStateInformation()` map trivially to
  serialize/deserialize the current snapshot. No translation layer.
- Audio thread code is minimal: `auto snap = store_.snapshot()` at top of
  `processBlock`. No queue draining, no partial-state consistency checks.
- User requirement: **everything configurable** — LLM bridge, API key, endpoint,
  profile. This separates into two distinct stores (see below).

---

## Two-store architecture

LAYERZ maintains **two separate stores** with different ownership and lifecycle:

### 1. `ProjectStore` — musical state (per .layerz file)

Holds the canonical `Project` model. Subject to lock-free-snapshot.

```cpp
class ProjectStore {
public:
    // Audio thread: O(1), no allocation, no lock
    std::shared_ptr<const Project> snapshot() const noexcept;

    // GUI/LLM thread: clones current, applies mutation, atomic pointer swap
    void postMutation(std::function<void(Project&)> fn);

    // Serialization (GUI thread only)
    juce::Result loadFromFile(const juce::File& f);
    juce::Result saveToFile(const juce::File& f) const;

private:
    std::atomic<std::shared_ptr<const Project>> current_;
};
```

`postMutation` clones the current snapshot, applies `fn`, then atomically swaps.
Clone cost is acceptable: Project is small (JSON-shaped struct, not audio buffers).
Copy-on-write expansion is deferred to later phases if clone cost proves significant.

### 2. `UserConfig` — installation settings (per user, per machine)

**Never stored in `.layerz` files.** Persisted in a platform-appropriate location:
- macOS: `~/Library/Application Support/LAYERZ/config.json`
- Linux: `~/.config/layerz/config.json`
- Windows: `%APPDATA%\LAYERZ\config.json`

```cpp
struct LLMConfig {
    bool    enabled       = false;          // opt-in; disabled by default
    std::string provider  = "anthropic";    // provider-neutral; first target
    std::string endpoint  = "";             // empty = use provider default
    std::string api_key   = "";             // never logged, never in .layerz
    int     timeout_ms    = 30000;
};

struct UserConfig {
    Profile     profile_override = Profile::AUTO;  // AUTO = infer from host context
    LLMConfig   llm;
    bool        show_splash      = true;    // JUCE GPLv3 splash — must not be hidden
    // add fields per phase; UserConfig is versioned (schema_version field)
    int         schema_version   = 1;
};
```

`UserConfig` is **read-only on the audio thread** (loaded at startup, reloaded on
user settings change via a separate atomic pointer swap — same pattern as `ProjectStore`
but mutations are rare). The audio thread never writes `UserConfig`.

### Configuration principles

- **LLM bridge is opt-in, disabled by default.** First launch: bridge is off.
  User explicitly enables it in settings, supplies their own API key.
  Key is stored in `UserConfig` (local machine only), never written to `.layerz`,
  never logged, never sent except to the configured endpoint.
- **Profile is overridable.** `Profile::AUTO` detects plugin vs standalone context
  at runtime. User can override to PLUGIN or STANDALONE per session.
  Override is stored in `UserConfig`, not in the `.layerz` project file —
  so the same project file runs at plugin caps in one context and standalone caps
  in another without modification.
- **All configurable values have safe defaults.** Fresh install: LLM off,
  profile AUTO, no API key. LAYERZ is fully functional without any configuration.
- **`UserConfig` is versioned** (`schema_version` field) for the same migration
  reasons as `Project`. Unknown fields are skipped on load.

## Implementation consequences for F0

`Source/model/Project.h` must expose:

```cpp
// Thread-safe store — wraps Project with lock-free snapshot semantics
class ProjectStore {
public:
    std::shared_ptr<const Project> snapshot() const noexcept;
    void postMutation(std::function<void(Project&)> fn);
    juce::Result loadFromFile(const juce::File&);
    juce::Result saveToFile(const juce::File&) const;
};
```

`Source/config/UserConfig.h` (new file, F0):

```cpp
struct LLMConfig { bool enabled; std::string provider, endpoint, api_key; int timeout_ms; };
struct UserConfig {
    int schema_version = 1;
    Profile profile_override = Profile::AUTO;
    LLMConfig llm;
    bool show_splash = true;
    static UserConfig load();   // reads platform config file; returns defaults on missing
    juce::Result save() const;  // atomic write (tmp + rename)
};
```

`AudioEngine` / `PluginProcessor` reads Project via:
```cpp
auto snap = project_store_.snapshot();  // at top of processBlock — O(1), no alloc
// use snap->patterns, snap->tempo, etc. for duration of block
```

GUI thread mutates Project via:
```cpp
project_store_.postMutation([](Project& p) {
    p.patterns[0].layers[0].events[step].velocity = newVelocity;
});
```

LLM bridge reads Project for API context via:
```cpp
auto snap = project_store_.snapshot();
auto json = projectToJson(*snap);  // clean, complete, ready to send
```

## What is NOT configurable (intentional)

- Number of layer types (always 4: BEAT, BASS, HARMONIC, MELODIC)
- Dark mode (always on in v1)
- JUCE splash screen (always shown — GPLv3 legal obligation; `show_splash` exists
  for future paid-license tier, not for user toggle in free build)
- Thread-safety model itself (lock-free-snapshot is architectural, not runtime config)
