# Spike F0.99 — Performance Budget

**Status:** CLOSED
**Date:** 2026-05-28
**Author:** bidu@lavid.ufpb.br

---

## Question

How many simultaneous voices and active grains can run at 256-sample, 512-sample,
and 1024-sample buffer sizes on each target machine while staying under the CPU
budget targets? What integer caps should be hardcoded in `ProfileConfig.h`?

---

## Method

`Source/bench/VoiceBench.cpp` — a standalone C++ benchmark (no JUCE audio device)
that simulates synthetic voice and grain workloads:

- **Voice proxy:** sine oscillator (frequency + phase advance per sample)
- **Grain proxy:** Hann-windowed sine at varying positions (approximates BLOOM/PULVERIZE grain reads)
- **Statistical validity:** 5 independent 10-second windows per configuration; reports mean + stddev across windows
- **Compiler flags:** `-O3 -DNDEBUG -std=c++17` (Release-equivalent)
- **Clock:** `std::chrono::steady_clock` (monotonic; avoids Windows `high_resolution_clock` non-monotonicity bug that produced negative measurements on first run)
- **Sweep:** voices 4–64 in steps of 4; grains 8–128 in steps of 8; buffer sizes 256, 512, 1024
- **Output:** TSV — `buffer_size | n_voices | n_grains | peak_cpu | mean_cpu | stddev_cpu | overruns`

CPU% is relative to the real-time block budget at 44100 Hz sample rate.

## Critical limitation: synthetic bench ≠ real DSP cost

**The synthetic bench severely underestimates real Faust DSP cost.**

| Component | Bench ops/sample | Real Faust ops/sample (estimated) | Est. multiplier |
|-----------|-----------------|-----------------------------------|-----------------|
| Voice (bench) | ~3 FP (sine + phase + accumulate) | ~20–80 FP (osc + envelope + filter + glide) | 7–25× |
| Grain (bench) | ~5 FP (windowed sine) | ~30–100 FP (buffer read + interp + window + pitch shift) | 6–20× |

The caps above use a **10× assumed multiplier** as the central estimate. This means:

- If real DSP is **cheaper than 10×** (simple voices, few partials): caps are overly conservative. Recalibration will loosen them.
- If real DSP is **more expensive than 10×** (complex granular, polyphony): caps need to be tightened. Recalibration will catch this.

**Recalibration is REQUIRED after F1** when the first real Faust drum voice and BLOOM skeleton exist. Methodology: replace synthetic voice/grain in a calibration build and re-sweep the same machines.

## Decision

### PROFILE_PLUGIN caps

```cpp
// Lenovo i7-8565U, buf=256: 12v+24g = 2.04% bench → ~20% real (10× est.)
// 4.6% headroom below 25% budget for OS jitter and measurement variance
static constexpr int MAX_VOICES_PLUGIN = 12;
static constexpr int MAX_GRAINS_PLUGIN = 24;  // BLOOM + PULVERIZE combined
```

Rationale: covers all 4 LAYERZ layers at realistic simultaneous voice counts
(4 drums + 1 bass + 1 bloom + up to 4 melodic = 10 max realistic, cap is 12).
24 grains is sufficient for BLOOM at organic-feel density (F5 will calibrate further).

### PROFILE_STANDALONE caps

```cpp
// Lenovo i7-8565U, buf=1024: 24v+40g = 3.63% bench → ~36% real (10× est.)
// 3.7% headroom below 40% budget
static constexpr int MAX_VOICES_STANDALONE = 24;
static constexpr int MAX_GRAINS_STANDALONE = 40;  // BLOOM + PULVERIZE combined
```

Rationale: 2× the voice cap and ~1.7× the grain cap vs plugin mode.
Larger buffer (1024 vs 256) provides proportionally more compute budget per callback.

## Consequences for downstream phases

| Phase | Impact |
|-------|--------|
| **F0** | `ProfileConfig.h` hardcodes the four integer constants above. Profile switching verified by F0-AC5. |
| **F1 (BEAT)** | Drum voice count cap: `min(4, MAX_VOICES_PLUGIN)` = 4. Within budget. |
| **F4 (HARMONIC/BLOOM)** | BLOOM grain cap = `MAX_GRAINS_PLUGIN / 2` = 12 in plugin mode (shared with PULVERIZE). Recalibrate here. |
| **F5 (PULVERIZE)** | PULVERIZE grain cap = `MAX_GRAINS_PLUGIN / 2` = 12 in plugin mode. Recalibrate here. |
| **F1 calibration** | After first real Faust voice exists: re-run VoiceBench with calibration shim, compare to synthetic baseline, adjust multiplier estimate, update caps if needed. |
