<objective>
## Goal
Answer two architectural questions that block all F0 implementation: (1) how many simultaneous voices and active grains are affordable at each buffer size on each target machine, and (2) which thread-safety pattern to use for the Project model between the audio thread and GUI/LLM threads.

## Purpose
F0.99 sets the hard caps for PROFILE_PLUGIN and PROFILE_STANDALONE — without these numbers, the profile switching logic has no values to switch between. F0.99b determines the fundamental audio/GUI communication pattern — every component that touches Project after this must follow the chosen approach. Both decisions are irreversible without significant rework.

## Output
- `docs/spikes/F0.99-performance-budget.md` — question, method, raw measurements table, chosen caps, consequences for downstream phases
- `docs/spikes/F0.99b-thread-safety.md` — question, options analysed, chosen pattern with rationale, consequences
- `Source/bench/VoiceBench.cpp` — benchmark harness (kept in repo; useful for regression testing later)
</objective>

- Real-time audio safety: no allocations, no locks, no file I/O on audio thread
- Platforms: Linux, macOS (Apple Silicon), Windows

<task type="checkpoint:decision" gate="blocking">
  <decision>Thread-safety model for Project model access between audio thread and GUI/LLM threads</decision>
  <context>
    The audio thread reads Project every processBlock() — it cannot block, cannot allocate, cannot lock.
    The GUI thread and LLM bridge write mutations to Project (step toggles, parameter changes, PatternPatches).
    Three patterns are viable:
  </context>
  <options>
    <option id="lock-free-snapshot">
      <name>Lock-free atomic snapshot (double-buffer)</name>
      <pros>Audio thread reads a frozen copy — zero contention, zero latency. Mutations are atomic pointer swaps. No lock on audio thread ever. Predictable worst-case.</pros>
      <cons>Two copies of Project in memory at all times. Mutations require cloning the whole model (or a CoW approach). Slightly stale reads possible (one block lag). More complex mutation path.</cons>
    </option>
    <option id="message-queue">
      <name>Message queue (GUI → audio via lock-free FIFO)</name>
      <pros>No full copy of Project needed. Mutations are small messages (change step N, set param X). Standard JUCE pattern (MessageManager, AudioProcessorValueTreeState uses a variant of this). Easier to reason about per-mutation.</pros>
      <cons>Audio thread must drain the queue per block — adds latency if queue is large. Parameter application must be atomic-enough per message. Slightly harder to snapshot "full state" for LLM bridge.</cons>
    </option>
    <option id="value-tree">
      <name>JUCE ValueTree with AudioProcessorValueTreeState</name>
      <pros>Off-the-shelf JUCE solution. Handles sync, undo, preset save/load. Well-tested. Reduces boilerplate.</pros>
      <cons>ValueTree is XML/dynamic-typed — less type-safe than a proper Project struct. Serialization to .layerz JSON requires a translation layer. May hide real-time safety issues behind JUCE abstractions. LLM bridge has to work through ValueTree API, not a clean JSON model.</cons>
    </option>
  </options>
  <resume-signal>Select: lock-free-snapshot, message-queue, or value-tree — and give one sentence of reasoning</resume-signal>
</task>
