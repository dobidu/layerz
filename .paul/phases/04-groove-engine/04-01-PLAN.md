<objective>
## Goal
Answer one architectural question that blocks DRAG/PUSH implementation: when a
micro-timing offset is applied to a beat event, what happens when the displaced
sample position falls outside the current audio block boundary (negative or > block_size)?

## Purpose
DRAG shifts events late (negative displacement). PUSH shifts events early (positive).
If the displaced position is < 0 or > block_size, the event either falls in the
PREVIOUS block (too late to render) or the NEXT block (must be carried forward).
The implementation strategy for DRAG/PUSH depends entirely on this decision.
Without this spike, the AestheticResolver in 04-02 will make the wrong choice.

## Output
- `docs/spikes/F3.99-groove-accuracy.md` — question, analysis, chosen pattern,
  implementation consequences for AestheticResolver
</objective>

## Key facts
- Clock emits BeatEvent{beat_index, sample_offset} where sample_offset ∈ [0, blockSize-1]
- BeatSequencer uses sample_offset to render voices at sub-block positions
- DRAG = negative displacement: sample_offset - drag_samples. May go < 0.
- PUSH = positive displacement: sample_offset + push_samples. May go ≥ blockSize.
- At 120 BPM, 256-sample buffer: one beat = 22050 samples ≈ 86 blocks per beat.
  Perceptual DRAG target: -10 to -30ms = -441 to -1323 samples. Exceeds single block.

  <action>
    Analyse the three strategies for handling out-of-block displaced offsets:

    **Strategy A — Clamp to block boundary**
    Clamp displaced offset to [0, blockSize-1]. Simple. Loses perceptual accuracy
    for large DRAG/PUSH values (everything > ~6ms clamped to block edge).
    Acceptable for small groove amounts; fails for dramatic lurch.

    **Strategy B — Carry-forward / carry-back queue**
    DRAG: if displaced_offset < 0, queue the event for the next block at
          blockSize + displaced_offset. Renders late in the NEXT block.
    PUSH: if displaced_offset ≥ blockSize, queue the event for the NEXT block
          at displaced_offset - blockSize.
    A small FIFO queue (max 1-2 entries) held in BeatSequencer carries pending events
    across block boundaries. Zero allocation after prepare().
    Perceptually accurate for any DRAG/PUSH value ≤ 1 block (~6ms at 256-buf).
    For values > 6ms: events chain across multiple blocks — still correct.

    **Strategy C — Pre-compute events per bar at bar start**
    At bar start, compile ALL events for the next 4 bars including micro-offsets,
    store in a pre-allocated buffer. Audio thread reads from the buffer, no real-time
    computation. Perfect accuracy, but requires bar-level pre-computation and
    resync on pattern change.

    **Analysis requirements:**
    1. What is the maximum DRAG/PUSH value the spec requires?
       F3-AC1: mean offset -10 to -30ms at 120 BPM. 30ms = 1323 samples = ~5 blocks.
       Strategy A fails here (clamps). Strategies B or C required.

    2. What is the implementation cost at F3?
       Strategy B: add a 4-entry carry queue to BeatSequencer. Small, correct, zero alloc.
       Strategy C: complex pre-computation, resync logic, larger scope.

    3. What does F3.99 need to validate?
       Test: at 256-sample buffer, emit BeatEvent at sample_offset=10, apply DRAG=-50 samples.
       Verify: the event renders at blockSize-40 in the NEXT block (strategy B result).
       Measure: timing error vs expected. Accept if < 1 sample.

    **Write docs/spikes/F3.99-groove-accuracy.md:**
    - Question (stated above)
    - Three strategies with analysis
    - Decision: Strategy B (carry-forward queue) — correct, bounded cost, no bar-level sync
    - Implementation consequences:
      - BeatSequencer: add `pendingEvents_` fixed-size queue (capacity: 8 events)
        cleared in prepare(), events from previous block dispatched first in process()
      - AestheticResolver: compute displaced sample_offset; if out of range,
        return a "deferred" marker so BeatSequencer puts it in pendingEvents_
      - Maximum supported DRAG/PUSH: any value ≤ 1 bar (4 beats × 22050 = 88200 samples)
        — effectively unlimited for perceptual groove purposes
